# Project rules lane

Use this only when the request changes project-wide principles, governance, or
the generated project rules that classic called a constitution workflow. Do not
create feature state for a rules-only change.

Inspect the current rule sources and their generated consumers. Preserve
unrelated custom rules and managed-block boundaries. Express each new rule as a
clear obligation with scope, rationale, conflict precedence, and a verifiable
effect on downstream workflows or artifacts.

Use the repository's deterministic rule renderer or context-update scripts when
available; do not hand-copy managed blocks across integrations. Verify every
generated consumer and report any integration that cannot represent the rule.
Treat a rule change affecting security, lifecycle, compatibility, or generated
protocols as a consequence-gate trigger.
