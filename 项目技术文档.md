# 项目技术文档

## 项目架构概览

`spec-kit-plus` 是一个以 Python CLI 为主入口的 Spec-Driven Development 工具仓库。核心入口位于 `src/specify_cli/__init__.py`，负责项目初始化、AI 集成安装、扩展管理、命令注册和运行时提示输出。

当前仓库可以分为四个主层次：

1. CLI 层：负责 `specify init`、`specify check`、`specify extension ...`、`specify team` 等用户入口。
2. Integration 层：位于 `src/specify_cli/integrations/`，负责不同 AI 集成的目录结构、命令模板转换和上下文文件生成。
3. Template / Script 层：位于 `templates/`、`.specify/templates/` 与 `scripts/`，负责初始化后的工作流模板和上下文更新脚本。
4. Verification 层：位于 `tests/`，通过 contract、integration 与 codex_team 测试保护 CLI、生成产物和跨集成边界。

### 本次功能发布边界

本次新增的是 Codex 专属的团队运行时能力，官方产品入口为 `specify team`。该能力在首个发布周期内遵守以下边界：

- 仅在 `specify init --ai codex` 路径中生成团队技能与运行时辅助文件。
- 仅在支持 `tmux` 的环境中提供可执行的运行时路径。
- 对既有 Codex 项目只提供可选升级支持，不作为首发阻塞项。
- 非 Codex 集成不得默认生成 `sp-team`、`.specify/codex-team/` 或宣传 `specify team`。

### 本次变更摘要

- 新增 `src/specify_cli/codex_team/` 作为 Codex 团队运行时的 Python 边界层。
- 新增 `src/specify_cli/runtime_vendor/` 作为嵌入式运行时子树与来源说明边界。
- 在 README、quickstart 和 changelog 中补充首发范围、隔离策略与维护说明。

## 目录结构及其职责

| 路径 | 职责 |
| --- | --- |
| `src/specify_cli/__init__.py` | 主 CLI 入口；聚合初始化流程、帮助输出、Codex 团队入口与扩展命令。 |
| `src/specify_cli/integrations/` | 各 AI 集成实现；负责生成 agent 命令、技能、prompt 与上下文文件。 |
| `src/specify_cli/integrations/codex/` | Codex 集成定义；扩展技能模板集合并接入 Codex 专属团队能力。 |
| `src/specify_cli/codex_team/` | Codex 团队运行时边界；包含命令文案、安装器、状态序列化、运行时桥接和状态路径逻辑。 |
| `src/specify_cli/runtime_vendor/` | 嵌入式 oh-my-codex 运行时子树与来源说明。 |
| `templates/commands/` | 共享命令模板；`team.md` 为 Codex-only 团队技能模板。 |
| `scripts/bash/`、`scripts/powershell/` | 上下文同步与项目辅助脚本。 |
| `tests/contract/` | 保护生成产物与 CLI 对外契约。 |
| `tests/integrations/` | 保护初始化流程、注册表行为与跨集成隔离。 |
| `tests/codex_team/` | 保护 Codex 团队运行时、升级路径与文档范围说明。 |
| `specs/001-codex-team-adapter/` | 当前特性规格、计划、任务拆解、快速验证说明与契约文档。 |

## 关键模块依赖关系图

```text
src/specify_cli/__init__.py
  -> src/specify_cli/integrations/__init__.py
  -> src/specify_cli/integrations/base.py
  -> src/specify_cli/integrations/codex/__init__.py
  -> src/specify_cli/codex_team/__init__.py
  -> templates/commands/*
  -> scripts/bash/update-agent-context.sh
  -> scripts/powershell/update-agent-context.ps1

src/specify_cli/codex_team/commands.py
  -> 向 CLI 暴露官方产品面 `specify team`

src/specify_cli/codex_team/installer.py
  -> 生成 `.specify/codex-team/*`
  -> 仅允许 Codex 集成安装团队辅助产物

src/specify_cli/codex_team/runtime_bridge.py
  -> 调用 runtime_vendor 运行时
  -> 校验 tmux 环境
  -> 写入运行时状态

tests/contract/*
  -> 约束 `specify team`、`sp-team` 与生成资产位置

tests/integrations/*
  -> 约束 Codex / 非 Codex 初始化行为边界
```

关键依赖原则如下：

- CLI 层通过 `codex_team` 子包聚合团队能力，而不是把运行时实现散落在主入口中。
- Codex 集成只在自身模板集合中加入 `team.md`，避免团队技能泄漏到其他 agent。
- 运行时桥接、状态路径、状态序列化分别拆分，保证状态协议可测试、可替换、可追踪。

## 核心类与接口功能说明

### `IntegrationBase` / `MarkdownIntegration` / `TomlIntegration` / `SkillsIntegration`

位置：`src/specify_cli/integrations/base.py`

职责：

- 抽象不同 agent 的模板安装流程。
- 统一 manifest 记录、模板复制和路径转换。
- 为 Codex/Kimi 等 skills 型集成提供 `SKILL.md` 产物写入能力。

### `CodexIntegration`

位置：`src/specify_cli/integrations/codex/__init__.py`

职责：

- 声明 Codex 的目录结构和上下文文件约定。
- 在共享技能列表之外注入 `team.md` 对应的 `sp-team` 技能。
- 保持团队能力只存在于 Codex 初始化路径。

### `codex_team.commands`

位置：`src/specify_cli/codex_team/commands.py`

职责：

- 暴露官方产品面常量 `TEAM_COMMAND_NAME` 和 `TEAM_SKILL_NAME`。
- 统一帮助文案、可用性提示和运行时状态摘要。
- 明确 `omx` 与 `$team` 不是本仓库的正式产品入口。

### `codex_team.installer`

位置：`src/specify_cli/codex_team/installer.py`

职责：

- 定义 `CODEX_TEAM_HELPER_FILES`。
- 在 Codex 项目中生成 `.specify/codex-team/runtime.json` 和 `.specify/codex-team/README.md`。
- 提供可选升级路径与缺失资产检查能力。

### `codex_team.manifests`

位置：`src/specify_cli/codex_team/manifests.py`

职责：

- 定义运行时会话与任务分发记录的数据结构。
- 提供 JSON 序列化/反序列化和状态载荷构造逻辑。

### `codex_team.state_paths`

位置：`src/specify_cli/codex_team/state_paths.py`

职责：

- 统一 `.specify/codex-team/` 下的运行时状态路径。
- 避免路径字符串散落在 CLI 与运行时桥接代码中。

### `codex_team.runtime_bridge`

位置：`src/specify_cli/codex_team/runtime_bridge.py`

职责：

- 校验 `tmux` 是否可用。
- 启动、分发、标记失败、清理运行时会话。
- 把嵌入式运行时与 Python CLI 之间的交互边界显式化。

## 核心数据流向图

### 流程一：`specify init --ai codex`

```text
CLI 参数
  -> 解析为 CodexIntegration
  -> 写入基础 Spec Kit 文件
  -> 安装共享技能 + `sp-team`
  -> codex_team.installer 写入 `.specify/codex-team/*`
  -> integration manifest 记录新增产物
  -> CLI 输出 `specify team` 的后续提示
```

### 流程二：`specify team`

```text
用户执行 `specify team`
  -> CLI 读取当前项目 integration
  -> codex_team.commands 生成帮助/可用性文案
  -> runtime_bridge 校验 `tmux`
  -> 读取或写入 runtime session / dispatch record
  -> 输出状态摘要与下一步建议
```

### 流程三：非 Codex 初始化隔离

```text
用户执行 `specify init --ai claude` 等非 Codex 流程
  -> 对应 integration 安装自身命令模板
  -> 不注入 `team.md`
  -> 不写入 `.specify/codex-team/*`
  -> CLI 帮助和初始化提示不宣传 `specify team`
```

| `specify team` | Codex-only 团队运行时入口；检查环境、展示状态并驱动运行时交互。 |

## `specify team` 运行面与守护

### 运行面

- `specify team status` 提供团队阶段、worker、task、mailbox 的 JSON 快照，方便排查阶段状态。
- `specify team await` 阻塞等待运行时进入 terminal phase，适合自动化脚本观察批次完成。
- `specify team resume` 读取 `.specify/codex-team/` 中的 session 元数据，重新启动 tmux 进程并恢复心跳。
- `specify team shutdown` 向运行时注册一个暂停请求，工作线程可以在完成当前任务后自然退出再记录事件。
- `specify team cleanup` 在 shutdown 成功后删除 `.specify/codex-team/` 状态目录，不要在攻击 shutdown 前运行。
- `specify team api <operation>` 通过结构化 JSON 操作（task claim、worker heartbeat、event append）访问 runtime 所分发的 API。

### 运行时状态存储

- `.specify/codex-team/runtime.json` 保存 session metadata 与 canonical state 路径。
- `.specify/codex-team/state/` 包含 `tasks/`、`workers/`、`mailboxes/`、`dispatch/`、`phase.json` 与 `events.log`，所有心跳、claim、审核、监控都写入这些文件。
- 状态目录采用 append-only 模式，CLI 在 resume、status、await、api 时只读取最新 JSON 记录，避免重复写入。

### 运维指导与后台要求

- 当前 runtime 依赖 `tmux`（Unix/WSL 或等效 Windows backend）作为 worker pane 运行时；`specify team` 运行前会校验 backend 是否可用。
- 使用 `specify team resume` 时必须确保 `.specify/codex-team/state/` 最新，避免同时启动多个 session。
- 在关闭流程中先运行 `specify team shutdown` 等待所有任务完成，再执行 `specify team cleanup` 清理文件，并在 `specify team await` 后确认 phase 结束。
- 如果需要观察特定阶段，可通过 `specify team status` 读取 `phase.json` 和 `events.log`，不要直接修改 state 文件。

### Codex-only 发布边界

- 只有 `specify init --ai codex` 会生成 `sp-team` skill 和 `.specify/codex-team/` 资产，其它 agent 不应触发该命令。
- 非 Codex 流程必须避免输出 `specify team` 相关文档、模板或 CLI hint，以免混淆安装者。
- `omx`、`$team` 等 alias 在本仓库不被视作产品面，所有官方文档与验证都以 `specify team` 为中心。

## 常见的代码模式与约定

1. 单一事实来源优先  
   例如 agent 配置来自 integration registry；Codex 团队帮助文件列表来自 `CODEX_TEAM_HELPER_FILES`。

2. 以显式边界隔离新能力  
   Codex 团队能力集中在 `src/specify_cli/codex_team/`，避免直接污染通用 integration 层。

3. 先用测试锁定边界，再扩展实现  
   contract 测试保护公开 CLI 与生成产物；integration 测试保护跨集成隔离；codex_team 测试保护运行时和升级路径。

4. 共享注册面视为协调点  
   `src/specify_cli/__init__.py`、integration 注册表、上下文更新脚本、模板目录都属于共享写面，修改时必须优先考虑非 Codex 回归。

5. 首发边界优先于功能扩张  
   当前版本只承诺新建 Codex 项目的团队运行时路径。不要在没有同步更新规格、README、quickstart、changelog 和测试的前提下把该能力扩展到其他 agent 或把可选升级改成强制升级。

6. 维护者注意事项  
   - 保持 `specify team` 作为正式产品入口，不要把 `omx` 或 `$team` 写回用户文档、帮助文案或验收步骤。  
   - 任何扩大支持范围的修改都必须同时更新 `tests/integrations/` 与 `tests/codex_team/`。  
   - 如果调整 `.specify/codex-team/` 产物结构，必须同步更新 installer、contract tests、quickstart 和 README。  
   - 可选升级路径应持续保持非阻塞语义，除非规格明确改变发布策略。
