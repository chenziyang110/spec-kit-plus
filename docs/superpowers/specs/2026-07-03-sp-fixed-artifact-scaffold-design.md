# SP Fixed Artifact Scaffold Design

## Purpose

Reduce `sp-*` workflow token cost by moving fixed, repeatable artifact content out of agent-authored replies and into deterministic scaffold scripts or copied templates.

The optimization target is workflow output cost, not prompt brevity alone. If an `sp-*` artifact is 30 KB and 10 KB is fixed structure, the fixed 10 KB should be generated by a script or copied from a template instead of being re-authored by the agent.

## Core Rule

Scripts may generate only fixed, mechanical content:

- Markdown frontmatter, fixed headings, fixed empty tables, fixed checklist structure, and fixed reviewer-guide text.
- JSON envelopes, required field names, default empty arrays or objects, version fields, and safe initial statuses.
- State-file skeletons where the fields and section order are deterministic.

Agents still own semantic work:

- Requirements, decisions, task decomposition, trade-offs, evidence interpretation, risk conclusions, readiness verdicts, and user-facing explanation.
- Any value that would require understanding intent, choosing scope, approving readiness, or resolving ambiguity.

Scripts must not fabricate `ready`, `approved`, `user_confirmed`, or equivalent state. Those fields may be initialized only to draft, pending, blocked, or empty values unless the caller supplies already-approved structured input.

## Non-Goals

- Do not rewrite all `sp-*` workflows in one pass.
- Do not move semantic judgment into scripts.
- Do not reduce quality gates to save tokens.
- Do not script artifacts whose fixed content is small or whose generated content would immediately need to be read back in full by the agent.

## Architecture

Add a small fixed-artifact layer behind the existing `specify` CLI and packaged generated-project scripts.

Two surfaces are needed:

1. `artifact audit-fixed-cost`
   - Read-only.
   - Scans `sp-*` workflow artifact contracts and known templates.
   - Reports fixed-content candidates, estimated savings, scriptability, quality risk, and recommendation.

2. `artifact scaffold`
   - Writes deterministic artifact skeletons.
   - Copies a template or builds a JSON envelope for a named `kind`.
   - Accepts a compact variables payload.
   - Emits compact JSON containing the written path, skipped/created status, estimated saved tokens, and `agent_fill_required` fields.

Example command shapes:

```text
specify artifact audit-fixed-cost --format json
specify artifact scaffold --kind quick-status --out .planning/quick/<id>-<slug>/STATUS.md --vars <json>
specify artifact scaffold --kind plan-contract --out <FEATURE_DIR>/plan-contract.json --vars <json>
```

The command names can be adjusted during implementation to fit existing Typer organization, but the behavior should remain split between read-only audit and artifact generation.

## Template Storage

Fixed Markdown scaffolds should live in a packaged template directory rather than inside long command prompts. JSON builders may either use small checked-in skeleton files or construct fixed envelopes in Python when that is clearer and easier to validate.

Generated-project availability matters. Any new templates or helper scripts must be included in package data and verified through `specify init` tests.

## Audit Model

Each candidate produces a compact record:

```json
{
  "workflow": "sp-quick",
  "artifact": "STATUS.md",
  "fixed_bytes": 9800,
  "semantic_bytes": 2400,
  "fixed_ratio": 0.8,
  "estimated_token_savings": 2450,
  "scriptability": "high",
  "quality_risk": "low",
  "recommendation": "scaffold",
  "agent_fill_required": ["current_focus", "scope_boundary", "validation_plan"]
}
```

Recommendation rules:

- `scaffold`: fixed content is substantial, deterministic, and low-risk.
- `builder`: machine-readable artifact benefits from schema validation and compact agent input.
- `skip_low_savings`: fixed content exists but token savings are too small.
- `skip_semantic`: content is mostly judgment, synthesis, or evidence interpretation.
- `defer_risk`: scriptability is plausible but quality or consumer behavior needs a narrower design.

The first implementation can use approximate token estimation such as bytes divided by four. Precision is less important than consistent ranking and clear savings evidence.

## First Candidate Set

Start with artifacts that are highly structured and repeatedly described in workflow prompts:

- `sp-quick`: `.planning/quick/<id>-<slug>/STATUS.md` fixed structure and safe initial state.
- `sp-debug`: `.planning/debug/<slug>.md` fixed session skeleton and checkpoint fields.
- `sp-discussion`: `handoff-to-specify.json` fixed envelope and Markdown/JSON consistency checks for required fields.
- `sp-specify`: `brainstorming/handoff-to-specify.json` compatibility skeleton.
- `sp-plan`: `plan-contract.json` fixed envelope.
- `sp-tasks`: `handoff-to-tasks.json`, `task-index.json`, and per-task packet skeletons where delegated execution needs fixed fields.

These are candidates, not automatic implementation commitments. The audit must confirm fixed ratio, savings, scriptability, and quality risk before each scaffold is introduced.

## Data Flow

For a scaffolded artifact:

1. The workflow gathers the minimum semantic facts needed to select the scaffold kind and fill safe variables.
2. The agent calls `artifact scaffold`.
3. The script writes only fixed structure and supplied safe variables.
4. The script returns compact JSON with `agent_fill_required`.
5. The agent edits or appends only semantic sections and values.
6. Existing self-review, validation hooks, and downstream consumers remain authoritative.

For JSON builders, the agent should pass compact structured input rather than prose. The builder fills the fixed envelope and validates required shape.

## Error Handling

- Do not overwrite existing artifacts unless the command receives an explicit force flag.
- Return `blocked_existing_file` when a target exists and force is absent.
- Return `invalid_vars` when caller input cannot satisfy the fixed scaffold's required variables.
- Return `unsafe_status` when caller input tries to set approval or readiness states that should only come from confirmed workflow gates.
- Return `unknown_kind` for unregistered scaffold types.

All default stdout should be compact JSON. Expanded diagnostics belong behind a debug or explain flag.

## Token Savings Assessment

Every scaffold change must include before/after evidence:

- Baseline artifact size and estimated fixed portion.
- Candidate generated fixed size.
- Agent-authored semantic size that remains.
- Estimated saved tokens per run.
- Whether the agent must read back the generated content.
- Quality-risk classification.

A scaffold should not ship when the saved tokens are marginal or when the generated artifact increases downstream reading cost enough to erase savings.

## Testing

Required tests:

- Audit tests for fixed-ratio calculation, recommendation labels, and skip behavior.
- Scaffold tests for each `kind`: created path, safe defaults, no-overwrite behavior, compact stdout, and `agent_fill_required`.
- JSON builder tests for schema shape and rejection of unsafe readiness or approval states.
- Integration tests proving generated assets are available after `specify init`.
- Workflow-template regression tests proving command guidance still names the same artifacts, quality gates, and downstream consumers.
- Packaging tests for any new templates, shared scripts, or Python modules.

## Rollout

1. Land the audit command and run it against all `sp-*` workflows.
2. Select the first low-risk scaffold candidates from audit evidence.
3. Add scaffold templates/builders for one or two artifacts.
4. Update only the affected workflow guidance to call the scaffold instead of asking the agent to write the fixed block.
5. Add tests and report measured token savings.
6. Repeat for the next candidates only when savings remain clear.

## Quality Bar

This change follows the repository's SP Workflow Compact Quality Standard. A candidate must reduce whole-chain cost while preserving behavior, intent, evidence quality, handoff consumability, downstream executability, validation closure, and residual-risk handling.

The hard quality target remains:

```text
candidate_quality_score / baseline_quality_score >= 0.98
```

If baseline quality is below 90, the candidate must improve absolute quality as well as reduce cost.

## Open Implementation Choices

These can be decided during implementation planning without changing the design:

- Whether scaffold templates live under `templates/artifacts/`, `.specify/templates/artifacts/`, or another packaged directory.
- Whether the Typer subcommand is nested under `artifact`, `workflow-artifact`, or an existing helper group.
- Whether the first JSON builders are table-driven skeleton files or Python constructors.
