# Two-Workflow Handbooks Runtime Atlas Design

**Date:** 2026-05-08
**Status:** Approved
**Owner:** Codex

## Summary

This design replaces the current layered runtime atlas with two
workflow-specific handbooks generated by `sp-map-build`:

1. `DEBUG-HANDBOOK.md` for `sp-debug`
2. `BUILD-HANDBOOK.md` for every major non-debug workflow

The core decision is to stop assuming that agents will traverse a layered
documentation tree just because prompts tell them to. Runtime atlas
consumption must match actual agent behavior: one workflow enters, one
handbook is required, and fixed chapter IDs define the minimum mandatory read
set before source reads or action planning begin.

This is a runtime atlas redesign, not a documentation polish pass. The old
`PROJECT-HANDBOOK.md`, `QUICK-NAV.md`, index JSON files, root topical docs,
and module trees exit the runtime consumption model and are no longer future
generated outputs for ordinary workflow use.

## Problem

The current project-map model assumes the following flow:

1. the workflow tells the agent to read the root navigation artifact
2. the agent notices which deeper docs might matter
3. the agent progressively traverses root, module, index, and deep workflow
   documents as needed

Downstream behavior shows that this assumption is wrong.

The product failure is not that workflows forgot to tell the agent to read
docs. The failure is that the required docs are fragmented across a layered
atlas that depends on agent initiative to traverse it.

Observed failure modes:

1. the agent often reads only the first layer and does not reliably descend
   into additional documents
2. important workflow-operational facts are spread across multiple runtime
   files, so the first document rarely contains enough action-ready guidance
3. the current atlas optimizes for information classification and navigation,
   not for single-pass workflow execution
4. `sp-debug` is especially harmed because successful debugging depends on
   quickly finding module boundaries, cross-module propagation paths, likely
   truth owners, and verification routes without relying on broad ad hoc repo
   exploration
5. `sp-specify`, `sp-plan`, `sp-tasks`, `sp-implement`, `sp-quick`,
   `sp-fast`, `sp-test-scan`, and `sp-test-build` are harmed because impact,
   collaboration, and verification facts are distributed across multiple
   topical docs instead of presented as one workflow-operational briefing

The result is a false sense of coverage: documentation exists, but the agent
does not reliably consume enough of it to make the right next decision.

## Goals

- Replace the layered runtime atlas with two workflow-specific runtime
  handbooks.
- Make `sp-debug` depend only on `DEBUG-HANDBOOK.md` for runtime atlas
  consumption.
- Make the other major brownfield workflows depend only on
  `BUILD-HANDBOOK.md` for runtime atlas consumption.
- Define fixed chapter IDs so each workflow can mandate a precise handbook
  read set instead of vaguely requiring "the atlas."
- Change `sp-map-build` success criteria from documentation breadth to
  workflow actionability.
- Ensure handbook content is task-oriented: where to start, what owns truth,
  what propagates, what to verify, and what is still unknown.
- Remove the expectation that agents will read multiple atlas files to gather
  basic operational context.
- Provide a direct way to prove runtime handbook consumption through prompt
  contracts, packet context bundles, and tests.

## Non-Goals

- Do not preserve compatibility with the current layered runtime atlas as an
  equally supported runtime mode.
- Do not keep generating `PROJECT-HANDBOOK.md`, `QUICK-NAV.md`,
  `.specify/project-map/index/*.json`, `.specify/project-map/root/*.md`, or
  `.specify/project-map/modules/**` for ordinary runtime workflow
  consumption.
- Do not optimize for human browseability at the expense of agent action
  readiness.
- Do not try to force a single universal handbook to serve both debugging and
  build/change workflows.
- Do not treat chapter references as soft guidance; they are part of the
  runtime contract.

## Key Terms

### Workflow Handbook

A single runtime atlas document that serves one workflow family directly and
contains the action-ready guidance needed before source inspection expands.

### Fixed Chapter ID

A stable section identifier inside a workflow handbook that a command prompt,
packet compiler, or validator can require explicitly. Workflow prompts bind to
chapter IDs, not to freeform prose locations.

### Runtime Handbook Consumption

The mandatory pre-source read of the relevant workflow handbook plus the
chapter IDs required for the current workflow tier before the agent begins
technical action such as debugging, planning, implementation shaping, or test
construction.

### Supporting Evidence

Live repository files, commands, logs, tests, or generated workbench artifacts
that may later confirm or refine a workflow handbook claim. Supporting
evidence is not a substitute for runtime handbook coverage.

## Approved Runtime Atlas Model

### 1. Runtime Atlas Outputs Collapse To Two Documents

`sp-map-build` should produce exactly two canonical runtime atlas outputs:

- `DEBUG-HANDBOOK.md`
- `BUILD-HANDBOOK.md`

These are the only future-generated documents that ordinary `sp-*` workflows
must treat as runtime atlas read surfaces.

### 2. Workflow Ownership Split

`DEBUG-HANDBOOK.md` serves only `sp-debug`.

`BUILD-HANDBOOK.md` serves the major non-debug workflow family, at minimum:

- `sp-specify`
- `sp-plan`
- `sp-tasks`
- `sp-quick`
- `sp-fast`
- `sp-implement`
- `sp-test-scan`
- `sp-test-build`

This is an explicit design choice. Debugging and build/change work have
different task models, different first questions, and different action
surfaces. They should not share one runtime manual.

### 3. Single-Handbook Runtime Consumption

Once a workflow enters runtime handbook mode, it must not assume that the
agent will traverse additional atlas files to find core guidance.

The workflow prompt may still permit live repository reads, logs, tests, or
other evidence-gathering after the handbook gate passes. What it must not do
is require the agent to read another atlas document to establish its base
mental model.

### 4. Layered Atlas Model Is Removed

The product is no longer built around:

- root navigation document
- machine index layer
- root topical document layer
- module document layer
- deep workflow subtrees

for runtime consumption.

Those surfaces are removed from the future-generated runtime contract rather
than preserved as parallel primary reads.

## Handbook Contracts

### `DEBUG-HANDBOOK.md`

Purpose:

- enable immediate bug investigation routing
- identify likely truth owners and cross-module failure paths
- show first inspection routes, failure branches, and fix verification routes
- prevent shallow symptom chasing that ignores module interaction

Required fixed chapter IDs:

1. `DEBUG-WORKFLOW-CONTRACT`
2. `SYMPTOM-TO-SURFACE-ROUTING`
3. `SYSTEM-TOPOLOGY-FOR-DEBUG`
4. `HOT-PATHS-AND-OWNERS`
5. `FAILURE-PATTERNS`
6. `INVESTIGATION-PLAYBOOKS`
7. `FIX-PROPAGATION-CHECKS`
8. `VERIFICATION-AND-EXIT`
9. `KNOWN-UNKNOWNS`

Chapter intent:

- `DEBUG-WORKFLOW-CONTRACT`: what `sp-debug` is allowed to do, what its gate
  is, and how handbook usage constrains the investigation
- `SYMPTOM-TO-SURFACE-ROUTING`: symptom families, likely modules, key entry
  files, and first inspection order
- `SYSTEM-TOPOLOGY-FOR-DEBUG`: cross-module causal routes and where failures
  commonly propagate
- `HOT-PATHS-AND-OWNERS`: likely truth-owning files, subsystems, and hot paths
- `FAILURE-PATTERNS`: recurring defect patterns, false leads, and likely
  failure mechanisms
- `INVESTIGATION-PLAYBOOKS`: concrete investigation scripts by bug type
- `FIX-PROPAGATION-CHECKS`: what must be re-checked after changing a truth-owning
  surface
- `VERIFICATION-AND-EXIT`: minimum fix proof and closeout expectations
- `KNOWN-UNKNOWNS`: areas where the handbook can only guide conservatively

### `BUILD-HANDBOOK.md`

Purpose:

- enable requirement shaping, planning, tasking, implementation shaping, and
  testing workflows from one action-oriented source
- identify where requested changes land in the repository
- identify change propagation, workflow sequencing, module collaboration, and
  verification routes

Required fixed chapter IDs:

1. `BUILD-WORKFLOW-CONTRACT`
2. `PRODUCT-AND-CAPABILITY-MAP`
3. `WORKFLOW-SEQUENCES`
4. `CHANGE-ENTRYPOINTS`
5. `MODULE-COLLABORATION`
6. `CHANGE-PROPAGATION-RISKS`
7. `IMPLEMENTATION-PLAYBOOKS`
8. `VERIFICATION-ROUTES`
9. `KNOWN-UNKNOWNS`

Chapter intent:

- `BUILD-WORKFLOW-CONTRACT`: which workflows the handbook serves and how each
  workflow consumes it
- `PRODUCT-AND-CAPABILITY-MAP`: the major product/runtime capabilities and
  where they live
- `WORKFLOW-SEQUENCES`: how `specify`, `plan`, `tasks`, `quick`, `fast`,
  `implement`, and test workflows relate
- `CHANGE-ENTRYPOINTS`: where to start for common change categories
- `MODULE-COLLABORATION`: how major code surfaces coordinate
- `CHANGE-PROPAGATION-RISKS`: where one change tends to ripple
- `IMPLEMENTATION-PLAYBOOKS`: practical work routes by task family
- `VERIFICATION-ROUTES`: minimum validation paths by change type
- `KNOWN-UNKNOWNS`: unresolved or lower-confidence areas that affect planning
  or execution

## Workflow Consumption Contract

### 1. The Atlas Gate Becomes A Handbook Gate

Current hard-gate language tells workflows to pass an atlas gate by reading
`PROJECT-HANDBOOK.md`, index files, root topic docs, module docs, and deep
workflow docs.

That model is replaced by a handbook gate.

Workflow prompts must no longer say:

- read the root navigation artifact
- choose the smallest relevant root topic set
- read the module overview
- read the deep workflow page

Instead they must say:

- read the required handbook
- read the required chapter IDs for this workflow
- block if handbook coverage is missing, stale, or insufficient for the
  current task

### 2. Fixed Chapter Sets By Workflow

Approved minimum chapter combinations:

#### `sp-debug`

Must read:

- `DEBUG-WORKFLOW-CONTRACT`
- `SYMPTOM-TO-SURFACE-ROUTING`
- `SYSTEM-TOPOLOGY-FOR-DEBUG`
- `INVESTIGATION-PLAYBOOKS`
- `VERIFICATION-AND-EXIT`

#### `sp-specify`, `sp-plan`, `sp-tasks`

Must read:

- `BUILD-WORKFLOW-CONTRACT`
- `PRODUCT-AND-CAPABILITY-MAP`
- `WORKFLOW-SEQUENCES`
- `MODULE-COLLABORATION`
- `CHANGE-PROPAGATION-RISKS`

#### `sp-implement`, `sp-quick`, `sp-fast`

Must read:

- `BUILD-WORKFLOW-CONTRACT`
- `PRODUCT-AND-CAPABILITY-MAP`
- `CHANGE-ENTRYPOINTS`
- `IMPLEMENTATION-PLAYBOOKS`
- `CHANGE-PROPAGATION-RISKS`
- `VERIFICATION-ROUTES`

#### `sp-test-scan`, `sp-test-build`

Must read:

- `BUILD-WORKFLOW-CONTRACT`
- `WORKFLOW-SEQUENCES`
- `CHANGE-PROPAGATION-RISKS`
- `VERIFICATION-ROUTES`

### 3. Command Tier Depth Moves Inside One Document

The current command-tier model controls how deeply the workflow must traverse
the layered atlas.

That depth model remains, but now applies to chapter depth inside one
handbook:

- `trivial`: minimum required chapter set only
- `light`: minimum chapter set plus relevant playbook/routing sections
- `heavy`: minimum chapter set plus all relevant collaboration, propagation,
  and verification chapters

The workflow no longer uses tier depth to justify loading more atlas files.

### 4. Handbook-Only Primary Read Rule

Workflow prompts must explicitly prohibit treating other atlas documents as
primary read surfaces.

The workflow may allow further repository evidence gathering after the handbook
gate passes, but must not say or imply:

- "read the smallest relevant atlas files"
- "continue through module docs"
- "use the navigation system to find deeper pages"

If a handbook section says an area is `Unknown` or needs live confirmation,
the agent may go to repository evidence next. It may not be forced through a
second atlas-doc discovery phase first.

## `sp-map-build` Generation Rules

### 1. Generate By Workflow Questions, Not By Repository Taxonomy

`sp-map-build` should not begin from root docs, module docs, and deep pages
and then hope a workflow can reconstruct a usable task route.

It must generate each handbook around workflow-operational questions:

- for debug: where does this symptom likely start, who owns truth, where does
  the failure propagate, what do I inspect first, how do I verify a fix
- for build/change: where does this requested change land, what does it touch,
  what is risky, what sequence matters, how do I verify it

### 2. Every Handbook Fact Must Answer A Workflow Decision

A fact belongs in a runtime handbook only if it helps answer a concrete next
step such as:

- where to begin
- what to inspect next
- what other surface is likely involved
- what can break if this changes
- what minimum verification must happen

Purely classificatory or encyclopedic content that does not improve a workflow
decision should not appear in the runtime handbooks.

### 3. Critical Facts Must Stay In-Handbook

Handbook content may cite live repository evidence, but must not externalize
critical runtime facts into another atlas document with "see elsewhere"
language.

The workflow handbooks are not thin portals. They are the runtime atlas.

### 4. Unknowns Must Be Workflow-Oriented

When the product cannot prove a fact confidently, the handbook must state:

- what is unknown
- which workflow decision is affected
- how the agent should proceed conservatively

This prevents a false appearance of certainty while still keeping the workflow
inside the handbook model.

### 5. Internal Structure Must Be Strongly Structured

The handbooks must not become narrative prose dumps.

Preferred internal units:

- routing tables
- owner cards
- propagation matrices
- playbook cards
- verification route cards
- known-unknown cards

This structure exists to keep handbook consumption scannable for agents and to
make chapter-level enforcement stable.

## Acceptance Criteria

The design is complete when all of the following are true:

1. `sp-map-build` runtime outputs are exactly `DEBUG-HANDBOOK.md` and
   `BUILD-HANDBOOK.md`.
2. `sp-debug` requires only `DEBUG-HANDBOOK.md` plus its mandated chapter IDs
   before debug-side source inspection begins.
3. The major non-debug workflows require only `BUILD-HANDBOOK.md` plus their
   mandated chapter IDs before planning, implementation shaping, or test-side
   source inspection begins.
4. Workflow prompts no longer instruct agents to consume
   `PROJECT-HANDBOOK.md`, `.specify/project-map/index/*.json`,
   `.specify/project-map/root/*.md`, or `.specify/project-map/modules/**` as
   runtime atlas reads.
5. Packet context bundles and default read scopes no longer inject the old
   atlas surface family as primary runtime context.
6. Both handbooks contain the fixed chapter IDs required by their workflow
   contracts.
7. Handbook content is action-oriented enough that a representative debug task
   and a representative build/change task can begin from the relevant handbook
   without needing a second atlas-document traversal phase.
8. Tests explicitly fail if the old layered atlas paths re-enter workflow
   prompts or primary packet context.

## Proof Of Real Consumption

The redesign must prove runtime handbook usage at three levels.

### 1. Prompt Contract Proof

Each workflow template must explicitly declare:

- required handbook path
- required chapter IDs
- read-before-act rule
- block behavior when handbook coverage is missing or stale

### 2. Packet And Handoff Proof

Where the product compiles packet context or subagent read bundles, the
primary injected context must come from the relevant handbook and chapter
selection rather than the old atlas file list.

### 3. Test Proof

Add or update tests so they assert:

- `sp-debug` points to `DEBUG-HANDBOOK.md`
- other major workflows point to `BUILD-HANDBOOK.md`
- old atlas paths are absent from runtime prompt contracts
- handbook chapter IDs are present
- packet context bundles no longer default to the old atlas family

Behavioral acceptance should include at least:

- one representative bug investigation that starts from
  `DEBUG-HANDBOOK.md`
- one representative requirement/change workflow that starts from
  `BUILD-HANDBOOK.md`

If either scenario still depends on a second atlas-document discovery phase,
the design has not succeeded.

## Migration Strategy

The approved migration is a strong cutover, not a long-lived parallel mode.

1. Redefine the `sp-map-build` runtime output contract around two handbooks.
2. Replace command-template atlas-gate language with handbook-gate language.
3. Remove old runtime atlas read requirements from command templates,
   passive skills, packet logic, and tests.
4. Rewrite docs and product guidance to teach the two-handbook model.
5. Delete or stop generating old runtime atlas artifacts rather than
   preserving them as an equally valid runtime mode.

The rationale for a strong cut is simple: if the old layered model remains
available as a parallel primary path, prompts and tests will drift back toward
it and reintroduce the exact behavior problem this design is solving.

## Risks and Mitigations

- **Risk:** the two handbooks become too large and agents still skim poorly.
  Mitigation: fixed chapter IDs, strongly structured cards and tables, and
  chapter-level workflow contracts.
- **Risk:** debug and build/change handbooks duplicate too much information.
  Mitigation: accept controlled duplication at the runtime layer; duplication
  is preferable to forcing runtime atlas traversal.
- **Risk:** `sp-map-build` becomes harder to synthesize.
  Mitigation: optimize for workflow-operational value, not encyclopedic
  completeness.
- **Risk:** the test surface is broad because many current tests encode the old
  atlas model.
  Mitigation: treat this as required migration work rather than incidental
  churn.
- **Risk:** a missing fact in one handbook can block an entire workflow family.
  Mitigation: surface `Known Unknowns` explicitly and make handbook
  insufficiency a clear block condition instead of hidden drift.

## Implementation Sequence

1. Redefine the `sp-map-build` output contract and runtime-success semantics
   around `DEBUG-HANDBOOK.md` and `BUILD-HANDBOOK.md`.
2. Define fixed chapter IDs and required chapter sets for each workflow
   family.
3. Replace atlas-gate language in command templates and shared partials with
   handbook-gate language.
4. Update packet compilation, validator, and hook logic to stop injecting old
   layered atlas files as primary runtime context.
5. Update docs and tests to enforce the two-handbook runtime model.
6. Run representative debug and build/change acceptance scenarios to verify
   that the workflow can act from the relevant handbook alone.
