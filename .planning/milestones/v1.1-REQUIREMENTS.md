# Requirements: spec-kit-plus

**Defined:** 2026-04-13
**Core Value:** Keep Spec-Driven Development practical across local AI integrations by making the workflow consistent, truthful, and usable in the tools developers actually run.

## v1 Requirements

### Workflow Routing

- [x] **WF-01**: `specify` can evaluate workload shape with the shared orchestration policy and record a `single-agent`, `native-multi-agent`, or `sidecar-runtime` decision before spec generation work begins.
- [x] **WF-02**: `plan` can evaluate workload shape with the shared orchestration policy and record a collaboration decision before research and design fan-out begins.
- [x] **WF-03**: `tasks` can evaluate workload shape with the shared orchestration policy before task decomposition and parallel batch derivation begin.
- [x] **WF-04**: `explain` can evaluate whether collaboration is justified and record the decision used for artifact explanation support work.

### Workflow Semantics

- [x] **LANE-01**: `specify` and `plan` define explicit lane purposes, join points, and fallback rules that match the multi-CLI collaboration design.
- [x] **LANE-02**: `tasks` and `explain` define workflow-specific lane purposes, join points, and conservative fallback rules appropriate to their work shapes.
- [x] **LANE-03**: `explain` remains conservative by default and only fans out for supporting cross-checks when the policy determines the extra work is justified.

### Integration Surface & Quality

- [x] **MSG-01**: Generated templates, CLI copy, and integration messaging keep native agent surfaces primary for non-Codex integrations and avoid advertising `specify team` outside Codex.
- [x] **QLT-01**: Collaboration-aware templates, README content, and CLI help use the same three strategy names and decision order across the expanded workflow surface.
- [x] **QLT-02**: Regression tests cover policy decisions, integration isolation, and generated asset language for `specify`, `plan`, `tasks`, and `explain`.

## v2 Requirements

### Runtime Maturity

- **RT-01**: `implement` supports deeper durable execution coordination on the shared orchestration core beyond the current release slice.
- **DBG-01**: `debug` can fan out evidence gathering in parallel while converging on a single writer before fix application.
- **BEND-01**: The portable process backend is production-ready across Windows-native and non-pane environments.

### Expansion

- **AGT-01**: Additional integrations beyond Codex, Claude, Gemini, and Copilot gain milestone-appropriate collaboration support.

## Out of Scope

| Feature | Reason |
|---------|--------|
| Full runtime maturity for `implement` | This milestone is limited to analysis and planning workflow expansion |
| Parallel code-modifying debug execution | Debug must stay more conservative until the runtime maturity milestone |
| New integration rollout beyond the current first adapter set | Expansion belongs to the later documentation and rollout milestone |
